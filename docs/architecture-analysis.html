✅ What Works Well

Strengths of the Current Implementation

Centralised Egress via Firewall

All spoke subnets have UDRs pointing 0.0.0.0/0 to the Firewall private IP. BGP propagation is explicitly disabled, preventing route override. The Firewall Policy is a separate resource — rule sets can be updated without recreating the Firewall.

Zero-Trust VM Access

No spoke VM has a public IP. Bastion Standard provides native client support and tunnelling. Audit logs are wired to Log Analytics. This eliminates the entire class of "exposed management port" incidents.

Private DNS Architecture

Private DNS zones for Storage and Key Vault are linked to both Hub and App VNets. PEP subnet has private_endpoint_network_policies = Disabled correctly set. DNS resolves PaaS FQDNs to private IPs.

Monitoring Depth

Six Firewall log categories, Bastion audit logs, NSG Flow Logs v2 with Traffic Analytics on both spokes, and three metric alerts with appropriate severities — all routed to a single Log Analytics workspace.

Dynamic Terraform Patterns

for_each on local.app_subnet_config with conditional UDR association (if v.apply_udr) avoids repetition without introducing unnecessary abstraction. The flat-root style keeps the dependency graph readable.

Secure Remote State

Blob Storage backend with TLS 1.2, public access disabled. Role-based access via az role assignment create in the bootstrap script. Secrets not stored in tfvars — Key Vault referenced at runtime.

🚨 Critical Gaps

Issues That Break the Design Intent

🛡️
NSGs created but have zero rules defined azurerm_network_security_group.app and .mgmt are empty. They are associated to every subnet purely for flow log capture. No East-West traffic is actually controlled — any VM in the app spoke can reach any other VM on any port. The NSGs provide an audit trail, not protection.
High
🌐
sn-dns subnet exists but nothing is deployed in it The Hub VNet has a dedicated DNS subnet (10.0.4.0/24). Its purpose is to host an Azure DNS Private Resolver or custom DNS forwarder. Without it, spoke VMs rely on Azure-default DNS (168.63.129.16) — this works for basic Private DNS lookups but breaks conditional forwarding to on-premises zones and provides no control over DNS query logging.
High
🔗
MGMT spoke not linked to Private DNS zones Only Hub and App VNets have azurerm_private_dns_zone_virtual_network_link entries. Jump VMs and tooling in the MGMT spoke (10.2.0.0/16) cannot resolve *.blob.core.windows.net or *.vaultcore.azure.net private endpoint FQDNs — they will fall back to public IPs, breaking the private connectivity model.
High
🔌
No Private Endpoint resources actually deployed The sn-pep subnet and private DNS zones are correctly prepared but no azurerm_private_endpoint resources exist. The architecture is ready but incomplete — there are no actual Storage or Key Vault private endpoints to demonstrate the pattern end-to-end.
Medium
🔍
Network Watcher name mismatch (import fragility) Azure auto-creates NetworkWatcher_<region> in NetworkWatcherRG, but the Terraform resource uses the project naming convention (nw-dev-sc). The comment says to import it — but the import path must match the Azure-generated name. Any new environment or region requires a manual import step before terraform apply, which is an operational pitfall.
Medium
⚠️ Firewall Rules

Overly Broad Rules & How to Tighten Them

Current — Too Permissive

# Source: all RFC-1918 private space source_addresses = ["10.0.0.0/8"] # Destination: anything on the internet destination_addresses = ["*"] destination_ports = ["80", "443"] # DNS: any resolver anywhere destination_addresses = ["*"] destination_ports = ["53"]

Any host in 10.0.0.0/8 — including future spokes, on-prem ranges, or VPN peers — can reach any internet destination on 80/443. DNS rule allows any resolver, including attacker-controlled ones (DNS exfiltration risk).

Wildcard Azure FQDNs

Application rule *.azure.com, *.microsoft.com covers a huge surface area. A compromised VM can reach any Azure service in any tenant — including data-plane endpoints outside this subscription.

Recommended — Least Privilege

# Per-spoke sources, not /8 source_addresses = ["10.1.0.0/16"] destination_addresses = ["*"] destination_ports = ["443"] # drop port 80 # DNS: point to known resolvers only destination_addresses = [ "168.63.129.16", # Azure DNS "1.1.1.1", # Cloudflare ]

Scoped Azure FQDNs

Replace wildcard rules with specific FQDNs needed by the workload:

  • management.azure.com — ARM API
  • login.microsoftonline.com — AAD auth
  • <account>.blob.core.windows.net — Storage
  • <vault>.vault.azure.net — Key Vault

Consider Firewall Premium for IDPS

Standard SKU has no Intrusion Detection. Premium adds signature-based IDPS, URL categorisation, and TLS inspection — critical if spokes host internet-facing workloads.

⚠️ Deprecated Feature

NSG Flow Logs → VNet Flow Logs

NSG Flow Logs are being deprecated by Microsoft in favour of VNet-level flow logs. Migration is recommended.

Current: NSG Flow Logs v2

Resource: azurerm_network_watcher_flow_log targeting azurerm_network_security_group. Requires one flow log per NSG. Captures traffic at the NSG evaluation layer only.

  • One resource per NSG — grows with the number of spokes
  • Does not capture traffic that bypasses NSG (e.g., within-subnet)
  • Deprecated — Microsoft targeting removal
  • Separate storage account required (cost overhead)

What Changes in Practice

Remove the two azurerm_network_watcher_flow_log resources and the azurerm_storage_account.flow_logs storage account. Replace with a single VNet flow log targeting the VNets directly. Log Analytics is still the destination — Traffic Analytics continues to work.

# New resource type resource "azurerm_network_watcher_flow_log" "app_vnet" { target_resource_id = azurerm_virtual_network.app.id # storage_account no longer required # when targeting a VNet }

🌐 Topology Comparison

Hub-and-Spoke vs. Azure Virtual WAN

Both achieve centralised connectivity. The right choice depends on scale, hybrid requirements, and operational complexity tolerance.

DimensionHub-and-Spoke (current)Azure Virtual WAN
Management modelSelf-managed VNets + peeringsMicrosoft-managed hub fabric
Peering limit500 per VNetThousands of spokes via vHub
Routing controlFull UDR control per subnetLimited — vHub routing tables
VPN / ExpressRoute integrationManual — VPN Gateway in HubBuilt-in, multi-link ER support
Transit connectivity (spoke-to-spoke)Requires Firewall or NVA in hubNative any-to-any via vHub
Firewall integrationAzure Firewall in Hub VNetAzure Firewall Manager + Secured Hub
CostPay for individual servicesvHub hourly + data processing fees
Terraform maturityFull azurerm coveragePartial — some Portal-only config

Verdict for This Project

With two spokes and a single region, Virtual WAN adds cost and complexity without benefit. Hub-and-Spoke is the right choice here. Virtual WAN becomes worth evaluating when spoke count exceeds ~15, when multi-region transit is required, or when managing ExpressRoute at scale.

⚠️ DNS Gap

What Should Be in sn-dns — and How

The sn-dns subnet (10.0.4.0/24) is allocated but empty. Two main options exist.

Option A — Azure DNS Private Resolver (recommended)

PaaS service — no VMs to patch. Deployed into the subnet with an inbound endpoint (receives DNS queries) and an optional outbound endpoint (conditional forwarding to on-prem).

No VM maintenance
HA by default (zone-redundant)
Supports conditional forwarding to on-prem
DNS query logs to Log Analytics
~$150/mo per resolver endpoint
No local caching (slight latency vs. VM)

Option B — Custom DNS VMs (Windows DNS / Unbound)

Two VMs in sn-dns act as forwarders. Forward private zone queries to 168.63.129.16; forward on-prem zones to on-prem resolver.

Full control — any DNS software
Local caching reduces latency
Works with legacy on-prem DNS
OS patching, HA configuration needed
NIC-level custom DNS must be set on VNets
Boot-time DNS chicken-and-egg on first deploy

Minimum Viable Fix (no resolver deployed yet)

Set dns_servers = ["168.63.129.16"] explicitly on all VNets in Terraform. This forces Azure-default DNS and ensures Private DNS zone lookups work consistently — even before a resolver is deployed. Without this, spoke VMs may inherit stale or region-incorrect DNS settings after DHCP renewal.

resource "azurerm_virtual_network" "app" { # explicit Azure DNS — guaranteed Private DNS resolution dns_servers = ["168.63.129.16"] }
⚠️ Missing Layers

DDoS Protection · Azure Policy · Defender for Cloud

DDoS Protection

Both public IPs (Firewall, Bastion) are unprotected. Azure provides basic DDoS mitigation for free, but DDoS Network Protection adds adaptive tuning, rapid response, and cost protection guarantees.

~$2,700/mo per plan Covers all public IPs in linked VNets

For non-production environments, basic is acceptable. For production with SLA commitments, Network Protection is justified.

Azure Policy

No guardrails prevent configuration drift. Without Policy, a future deployment can introduce:

  • VMs with public IPs in spokes
  • Storage accounts without TLS 1.2
  • Resources missing required tags
  • Subnets without NSG associations
  • Unapproved regions for deployment

Policies can be assigned at subscription or management-group level and enforced via azurerm_policy_assignment in Terraform.

Microsoft Defender for Cloud

No Defender plans are enabled. Defender for Servers, Key Vault, Storage, and DNS provide runtime threat detection beyond what the Firewall and NSG logs expose.

  • Defender for Servers — MDEAV, JIT access
  • Defender for Key Vault — exfiltration alerts
  • Defender for Storage — malware scanning
  • Secure Score — continuous posture assessment

📋 Roadmap

Prioritised Improvement Backlog

1
Add NSG security rules Define inbound/outbound rules for web, app, and PEP subnets. Separate NSGs per tier for microsegmentation. Currently zero rules = zero East-West control.
2
Add Private DNS link for MGMT VNet One-line fix: add an azurerm_private_dns_zone_virtual_network_link for azurerm_virtual_network.mgmt. Jump VMs need private DNS resolution.
3
Tighten Firewall rules Scope source from 10.0.0.0/8 to per-spoke CIDRs. Replace wildcard DNS destination with known resolvers. Remove port 80. Scope Azure FQDNs to specific endpoints.
4
Migrate NSG flow logs → VNet flow logs Replace azurerm_network_watcher_flow_log (NSG target) with VNet-targeted resources. Remove the flow logs storage account. Reduces resource count and future-proofs monitoring.
5
Deploy Azure DNS Private Resolver into sn-dns Activates the reserved subnet. Enables conditional forwarding to on-prem zones and DNS query logging. Use inbound endpoint IP as dns_servers on all spoke VNets.
6
Deploy at least one Private Endpoint Add a Key Vault or Storage private endpoint in sn-pep with a DNS A-record in the existing zones. Completes the private connectivity model end-to-end.
7
Fix Network Watcher naming / import Either adopt the Azure-generated name in Terraform or use a data source for the Network Watcher. Remove the manual import requirement from the operational runbook.
8
Add Azure Policy assignments Enforce: no public IPs on spoke NICs, required tag set, NSG on all subnets, TLS 1.2 on storage. Prevents configuration drift as the platform grows.
9
Enable Defender for Cloud (Key Vault + Storage) Low-cost Defender plans for Key Vault and Storage provide threat intelligence that flow logs alone cannot surface (exfiltration patterns, anomalous access).
10
Evaluate Firewall Premium for IDPS If workloads are internet-facing or compliance requires IPS, upgrade. Otherwise Standard is cost-appropriate. Revisit when first spoke hosts a public-facing service.
