🎯 Project Goals

What This Architecture Solves

Problem Statement

Workloads spread across multiple VNets need a unified, auditable path to the internet and to shared PaaS services — without exposing individual VMs or services to the public internet.

Design Constraints

  • No public IPs on workload VMs
  • All egress must be inspected and logged
  • PaaS services must not traverse the public internet
  • Full infrastructure reproducibility via IaC
  • Separate network domains per workload type

Chosen Approach

Hub-and-spoke topology with a centralised Hub VNet hosting all shared security services. Spokes are isolated workload VNets peered to the Hub, with User-Defined Routes forcing all traffic through Azure Firewall.

Technology Stack

MS Azure Terraform 4.x Azure CLI Bash Azure Firewall Azure Bastion VNet Peering Private DNS Log Analytics NSG Flow Logs Key Vault Blob Storage
🏗 Architecture

Hub-and-Spoke Network Topology

Centralised security, routing and access management across multiple workload VNets.

Hub VNet

Central security perimeter. Hosts Firewall for egress, Bastion for zero-trust VM access, Shared Services and DNS resolver subnets.

Spoke VNets

Workload isolation. App spoke runs web/app tiers + private endpoint subnet. Mgmt spoke hosts jump and tooling VMs.

Traffic Flow

All spoke egress is UDR-forced through the Firewall. Bidirectional VNet Peering with allow_forwarded_traffic.

⚙ Azure Services

Key Components in Depth

🔥

Azure Firewall (Standard)

Central egress control for all spoke VNets. Configured with a Firewall Policy containing Network Rule Collections (allow HTTP/S, DNS from 10.0.0.0/8) and Application Rule Collections (*.azure.com, *.microsoft.com). All traffic from spokes is UDR-forced here.

Standard SKU Firewall Policy SNAT
🔐

Azure Bastion (Standard)

Agent-free, browser-based RDP/SSH access to all spoke VMs — zero public IPs on VMs. Standard SKU enables features like IP-based connection, native client support, and tunnelling. Audit logs streamed to Log Analytics.

Standard SKU No public VM IPs Audit Logs
🌐

Private DNS + Private Endpoints

Private DNS zones for privatelink.blob.core.windows.net and privatelink.vaultcore.azure.net. Zones linked to Hub and App VNets — PaaS services resolve to private IPs, never traversing the public internet.

Storage Key Vault VNet Links
🔑

Remote State + Key Vault

Terraform state stored in Azure Blob Storage with state locking. Secrets managed by Azure Key Vault. Backend bootstrapped via prepare-dev.sh — Azure CLI script that creates the storage account and vault before Terraform runs.

Blob Backend State Lock Key Vault

Style Guide

Follows carlzxc71/lindbergtech flat-root convention — complex resources separated by domain into dedicated .tf files, not sub-modules. Variables: description first, type second. Naming: <abbr>-<env>-<region>-<descriptor>.

# Dynamic subnet creation with conditional UDR locals { app_subnet_config = { "sn-web" = { cidr = "10.1.1.0/24", udr = true } "sn-app" = { cidr = "10.1.2.0/24", udr = true } "sn-pep" = { cidr = "10.1.10.0/24", udr = false } } } resource "azurerm_subnet" "app" { for_each = local.app_subnet_config name = each.key address_prefixes = [each.value.cidr] resource_group_name = azurerm_resource_group.hub.name virtual_network_name = azurerm_virtual_network.app.name } # UDR: force all egress through Firewall resource "azurerm_subnet_route_table_association" "app" { for_each = { for k, v in local.app_subnet_config : k => v if v.udr } subnet_id = azurerm_subnet.app[each.key].id route_table_id = azurerm_route_table.app.id } # Bidirectional VNet Peering resource "azurerm_virtual_network_peering" "hub_to_app" { allow_forwarded_traffic = true allow_gateway_transit = false use_remote_gateways = false }

Egress Control

  • All spoke traffic → Firewall via UDR
  • HTTP/HTTPS + DNS allowed to Internet
  • Azure Management FQDNs whitelisted
  • All other traffic — implicit deny

PaaS Security

  • Private Endpoints for Storage + KV
  • Public blob access disabled
  • TLS 1.2 minimum on storage
  • Private DNS resolves to internal IPs

Network Segmentation

  • NSG per spoke for East-West control
  • PEP subnet: network policies disabled
  • Dedicated DNS subnet in Hub
  • Separate CIDR blocks per VNet

Certificates & TLS: Storage accounts enforce minimum TLS 1.2 at platform level. Key Vault hosts certificates with role-based access. Azure Firewall terminates TLS for application-layer inspection. Bastion uses Azure-managed HTTPS with valid public certificates — no self-signed exposure.

📊 Monitoring

Native Azure Observability Stack

Log Analytics Workspace

Central sink for all telemetry. 6 Firewall log categories — Network Rules, Application Rules, NAT Rules, Threat Intel, IDPS Signatures, DNS Proxy. Bastion audit logs. NSG Flow Logs with Traffic Analytics (10-min intervals). 30-day retention.

NSG Flow Logs v2

Enabled on App and Mgmt spoke NSGs. Stored in dedicated storage account (stdevscfl) with LRS. Traffic Analytics gives visibility into East-West and North-South flows, top talkers, and anomalies.

Action Group → Email Alerts

🧩 Design Decisions

Key Architectural Choices

Hub-and-Spoke vs. Flat VNet

A single flat VNet would scale poorly and share blast radius across workloads. Hub-and-spoke isolates each workload into its own VNet while centralising the expensive shared services (Firewall, Bastion, DNS) in the Hub — one set of resources, multiple spokes.

Firewall Standard vs. Premium

Standard SKU covers FQDN filtering, network rules, and full diagnostic logging — sufficient for this architecture. Premium adds TLS inspection and IDPS, appropriate when the firewall must act as a full SSL proxy for internal east-west traffic.

Private Endpoints vs. Service Endpoints

Service Endpoints keep traffic on the Azure backbone but the service still has a public IP. Private Endpoints inject the service directly into the VNet with a private IP — DNS resolves to a private address, eliminating any public exposure.

Flat Terraform Root vs. Nested Modules

Nested modules add indirection and complicate state. A flat root with domain-separated .tf files (firewall.tf, dns.tf, monitoring.tf) keeps the dependency graph visible and makes cross-resource references straightforward.

Bastion Standard vs. Basic

Basic SKU only supports browser-based RDP/SSH via Azure Portal. Standard adds native client support, IP-based connections, and tunnelling — needed for tooling that expects a real SSH session rather than a browser terminal.

Remote State in Blob + Key Vault

State stored in Azure Blob Storage with native lease-based locking prevents concurrent applies. Secrets referenced via Key Vault rather than hardcoded in tfvars — the backend is bootstrapped once with a Bash script before Terraform runs.
